# Domain — Hierarchy (Stage 3c)

Identify super-domains: groupings of sibling domains that share a parent concern. LLM-driven, low-frequency.

**Phase.** Stage 3c. **Experimental in v1.**

**Input → Output.** Sibling domain summaries within a path + `_index.yaml` → proposed tree edits applied to `_index.yaml` + cascading summary regeneration via [domain-semantic](../domain-semantic/spec.md).

---

## Status note

This stage is **experimental** in v1. The consultant wants to explore whether hierarchical organization adds enough value to be worth the LLM cost at this scale (~35 microservices). Expect this spec to evolve as the experiment runs.

## Behavior

- `identify_super_domains`: given sibling domain summaries within a path, propose groupings.
- Output is a proposed tree edit applied to `_index.yaml`.
- Re-summarization of new parents cascades but is cached, so unchanged subtrees cost nothing.

## Proposal → review → apply workflow

Hierarchy mutations follow a **proposal → review → apply** workflow, mirroring [taxonomy discovery](../taxonomy/spec.md):

1. **Propose.** `python -m assessment hierarchy:propose` runs `identify_super_domains` over sibling domain summaries and writes `domains/_hierarchy_proposal.md` — a human-readable document showing which sibling domains would be grouped, the proposed super-domain name, and the rationale.
2. **Review.** The consultant reviews and edits the proposal (accepting, rejecting, or renaming proposed super-domains). Rejected groupings are deleted from the proposal.
3. **Apply.** `python -m assessment hierarchy:apply` reads the (possibly edited) proposal and mutates `_index.yaml` accordingly.

This adds ceremony but catches bad merges before they cascade through summary regeneration. Given the experimental nature of this stage, the consultant should see what the LLM proposes before it restructures the domain tree.

## Tree mutation rules

- Super-domains are added by inserting new entries in `_index.yaml` with `origin: super` and the sibling domains' `parent` fields updated to point to the new super-domain.
- A super-domain's summary is generated by [domain-semantic](../domain-semantic/spec.md) on the next run; its inputs are the child domains' summaries plus their member counts and top extractions.
- Super-domains do not themselves contain member docs directly. Member docs always live in leaf domains.

## Cache key

- `hash(sorted(child_summary_hashes) + prompt_version + model)`.
- Unchanged subtrees produce identical proposals, so re-runs are free.

## Failure modes

- **Over-merging unrelated subsystems.** The LLM proposes a super-domain that groups things that share vocabulary but not concern. Mitigation: see clarification above — likely human review at hierarchy proposal time.
- **Under-merging genuinely related domains.** The LLM is conservative and misses sensible groupings. Acceptable in v1; the consultant can manually edit `_index.yaml`.
- **Cascading invalidation cost.** A change at a deep node forces re-summarization up the tree. Mitigation: hash-keyed caching means only the affected ancestor chain regenerates; siblings are untouched.
